# 장부 에이전트 — 프로젝트 규칙

소상공인 매입/판매 장부 + AI 매입 가이드 에이전트.
구조: Agent = Model + Harness. 모노레포 (backend/, frontend/).


## 절대 규칙 (모든 코드에 적용)
- 숫자 계산·집계·예측은 전부 결정론적 코드(Harness)가 한다. LLM은 그 결과를 해석/조언만.
- 응답·가이드에 들어가는 모든 수치는 반드시 툴 실행 결과에서만 온다. 코드가 숫자를 추정/생성하지 않는다.
- 아키텍처/스택/스키마를 바꾸는 결정은 먼저 사람에게 묻고 진행한다.
- 새 기능은 항상 테스트와 함께 만든다.

## 스키마 규칙
- `db/schema_ddl.sql`이 유일한 스키마 진실 공급원(Single Source of Truth)이다. 사용자가 항상 갱신·관리하며 신뢰한다.
- `db/db_dump.sql`은 공통 코드 초기 데이터의 진실 공급원이다.
  코드값(trxType 등)을 서비스 로직에서 참조하기 전, 반드시 이 파일에서 실제 code 값을 확인한다.
- ORM 엔티티 등록 및 CRUD 구현 전, 반드시 DDL에서 다음을 확인한다:
  - 컬럼 타입·길이, `NOT NULL` / `NULL` 여부, `DEFAULT` 값
  - `PK` / `FK` (참조 테이블, `ON DELETE` 동작)
  - `UNIQUE` 제약 (단일 컬럼 및 복합)
  - 인덱스 (조회 조건에 맞는 인덱스 존재 여부)
- ORM 엔티티는 DDL을 정확히 반영한다. DDL에 없는 컬럼을 엔티티에 추가하지 않는다.
- CRUD 로직은 DDL 제약조건을 위반하지 않는다:
  - `NOT NULL` 필드에 null 저장 금지
  - `UNIQUE` 충돌은 서비스 레이어에서 사전 검증 또는 예외 처리 필수
  - `FK` 참조 무결성 — 부모 레코드 존재를 보장한 뒤 자식 레코드를 삽입한다
- 스키마를 변경해야 하면 DDL을 먼저 수정하고, 사람에게 승인받은 뒤 엔티티·마이그레이션을 반영한다.

## 작업 방식
- 작업 전: 변경 계획을 먼저 제시하고 승인받은 뒤 진행한다.
- 작업 후: 사람이 검토·승인하기 전까지 다음 단계로 넘어가지 않는다.
- 한 번에 여러 단계를 진행하지 않는다. 한 단계 = 한 검토 = 한 커밋.

## 빌드 순서
Tool(결정론적 코어) → Guardrail → Eval → Observability → LLM 연결.
LLM은 가장 마지막에 얇게 붙인다. 먼저 꽂지 않는다.
