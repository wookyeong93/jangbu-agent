# ADR-0003: user 삭제 시 장부 데이터 처리 방식

## 맥락
- user 와 ledger 는 1:N 관계
- 회계성 데이터라 일반적으로는 감사 추적·정산 이력 목적상 거래기록을 보존하는 경우가 많음
- 다만 현재는 개인 토이 프로젝트로, 외부 정산·세무·다중 사용자 감사 요건이 없음
- user 탈퇴 시 그 user 의 장부 데이터를 보존할지(soft delete) 함께 제거할지(cascade) 결정 필요

## 결정
- user : ledger 관계에 `ON DELETE CASCADE` 적용. user 삭제 시 해당 장부 전체를 물리 삭제하며, 거래기록을 별도로 보존하지 않음.

## 고려한 대안
1. **ON DELETE CASCADE (채택)**
   - 구현 단순, 고아 레코드 없음, 정합성 관리 부담 없음
2. **Soft delete (deleted_at 컬럼) + 보존**
   - user/ledger 에 삭제 플래그만 찍고 데이터 유지
   - 감사·복구 가능하나, 모든 조회 쿼리에 `deleted_at IS NULL` 조건 누락 리스크 상시 존재
3. **삭제 시 ledger 별도 아카이브 테이블로 이관**
   - 운영 데이터에서 분리하면서 이력 보존
   - 이관 로직·스키마 이중 관리 비용 발생

## 결과
- (+) 스키마 삭제 로직 단순 , 토이 프로젝트 범위에 적합 
- (+) 조회 쿼리에 삭제 필터 조건 불필요 
- (-) 탈퇴 후 거래기록 복구 불가, 감사 추적 불가 
- (-) 향후 정산·세무·다중 사용자 요건 생기면 이 결정 재검토 필요 (보존 정책으로 마이그레이션 비용 발생)
