# ADR-0001: 영속성 프레임워크 분리 (JPA/QueryDSL + MyBatis)

## 맥락
장부 도메인은 두 종류의 데이터 접근이 섞여 있다.
- 매입/판매/지출 입력 같은 단순 CRUD, 조건별 동적 조회
- 월별·요일별 집계, 순익 계산 등 윈도우 함수·복잡한 group by가 필요한 통계 쿼리
단일 프레임워크로 둘 다 처리하면 한쪽이 불편해진다. JPA로 복잡한 통계 쿼리를 짜면 가독성·성능 제어가 떨어지고, MyBatis로 단순 CRUD를 짜면 보일러플레이트가 늘어난다.

## 결정
- 단순 CRUD·타입세이프 동적 쿼리 → JPA + QueryDSL
- 복잡한 통계/집계 쿼리 → MyBatis (수기 SQL)
- 하네스 Tool 레이어에서 "정확한 수치"를 책임지는 집계 쿼리는 MyBatis로 작성한다.

## 고려한 대안
- JPA 단독: 통계는 네이티브 쿼리/JPQL로 우회 → 가독성·유지보수 저하
- MyBatis 단독: 단순 CRUD까지 수기 SQL → 보일러플레이트 증가

## 결과
- (+) 각 도구를 강점에 맞게 사용. 통계 쿼리 가독성·성능 확보, CRUD 생산성 유지
- (−) 영속성 프레임워크 2개 운영 → 학습·유지 비용 증가, 엔티티↔수기 SQL 결과 매핑 일관성 주의
